# 001 — 开源 ASR 对比评测与选型执行计划

## 1. 背景与目标

在固定测试语料上对比多款**开源**自动语音识别（ASR）方案，量化**推理耗时/实时率**与**转写错误率**，筛选在准确率与速度上满足产品预期的实现，作为本仓库 `speech-to-text` 后续集成的候选。

**非目标（本阶段不做）**：线上服务化改造、流式低延迟方案专项优化、多说话人分离。

## 4. 评测指标

### 4.1 性能

- **墙钟时间**：单文件从「读入音频到输出文本」的总耗时（秒）。
- **音频时长**：转写对应音频的时长 \(T\)（秒）。
- **RTF（实时率）**：\(\mathrm{RTF} = \frac{\text{墙钟时间}}{T}\)；越小越好（小于 1 表示快于实时）。

**记录项**：硬件型号（例如 Apple M 系列）、是否使用 GPU/Metal/CoreML、模型名称与尺寸、线程/批大小等关键参数。

### 4.2 错误率

- **中文**：优先 **CER（字错误率）**；若分词稳定可补充 **WER**。
- **英文或中英混合**：**WER** 为主。

**前提**：必须有与每条音频对齐的**参考全文**（人工听写或已有字幕）。没有参考文本则**无法**计算 CER/WER，只能报告主观听感或定性对比。

**建议流程**：

1. 由听音员（或项目负责人）整理 `testspeed` 每条音频的参考文本，存为 UTF-8 纯文本（如 `1.ref.txt` 与 `1.mp3` 同名不同扩展名，或集中放在一个 `refs/` 目录并约定命名规则）。
2. 对模型输出做统一后处理：去除首尾空白、全半角与标点策略在全部系统间保持一致（必要时仅保留中文汉字与英文单词用于对齐）。
3. 使用同一脚本计算 CER/WER（例如 Python `jiwer` 或自研 Levenshtein，中文 CER 按字符切分）。

## 5. 环境与可复现性

- **OS**：macOS（与 README 一致，Apple Silicon）。
- **记录**：Python/Conda、pip 或 `uv` 锁版本；Go 版本；各模型权重文件的版本或下载 URL、SHA256（若可取得）。
- **冷启动与热启动**：首次加载模型时间可单独记录；主表格以「模型已加载后」的单次推理为准，避免把下载权重时间算进 RTF。

## 6. 执行步骤（分阶段）

### 阶段 0：准备

- 确认 `testspeed` 文件列表与 checksum；完成参考文本收集与审阅。
- 确定统一音频预处理脚本与输出目录结构（例如 `artifacts/{run_id}/`）。

### 阶段 1：基线跑通

- 在本仓库现有 **whisper.cpp（C）** 路径上跑通 4 条音频，记录耗时与转写结果快照。
- 用同一参考文本计算 CER/WER，验证评测脚本正确（可用手工篡改一句参考文验证指标是否敏感变化）。

### 阶段 2：Python 栈对比

- 安装并固定 **openai-whisper** 与 **faster-whisper** 版本；选用同一套 Whisper 权重规格（如 `small` 或 `medium`）以便横向可比。
- 可选：在 Apple Silicon 上尝试 faster-whisper 的 device 设置（CPU / 若支持则 Metal/CoreML），分别记录。

### 阶段 3：中文增强栈

- 部署 **FunASR**（或当前官方推荐的 Paraformer / 说话人无关通用模型），用相同语料评测。
- 若环境允许，增加 **Sherpa-ONNX** 一条非流式模型作为 ONNX 路线对照。

### 阶段 4：轻量对照

- 运行 **Vosk** 合适的中文/多语言模型，记录速度与错误率（权重体积通常较小）。

### 阶段 5：汇总与选型

- 汇总表格：系统 × 模型 × 每条语料的 RTF 与 CER/WER。
- **准入建议（需团队确认阈值）**：例如在参考集上 CER 不高于某值且 RTF 满足产品上限；未达标方案标注「不保留」及原因。
- 对「保留」方案列出集成到本仓库的成本（依赖、许可证、二进制体积、维护频率）。

### 选型决策表（模板）

评测结束后按行打分（示例维度，权重由团队调整）：

| 方案 | 平均 RTF | 平均 CER | Apple Silicon 体验 | 依赖与维护 | 是否保留 |
|------|----------|----------|--------------------|------------|----------|
| … | … | … | … | … | 是/否 |

**保留规则建议**：先按 CER 排序取前两名，再在二者中比较 RTF 与集成成本；若 CER 接近（例如相对差异小于约定百分比），优先 RTF 更低者。

## 9. 风险与应对

| 风险 | 应对 |
|------|------|
| 依赖冲突（多 Python 环境） | 每栈独立 venv/conda env；README 中记录 env 名称 |
| 无参考文本 | 暂停错误率维度，先只报 RTF；或缩小为先完成 1～2 条人工标注 |
| Apple Silicon 上某框架 GPU 路径不可用 | 明确记录为 CPU 结果，并在表格中标注 |
| 模型下载失败或体积过大 | 优先较小权重完成全链路；大模型作为后续迭代 |
